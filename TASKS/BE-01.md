## 🎯 Summary
- 기능명: [BE-01] 수동 파일 업로드 수신 및 FastAPI 바이너리 전달 API 구성 (`POST /api/v1/clean` 역할)
- 목적: 프론트엔드로부터 바이너리 파일(법률 문서)을 업로드 받아 1차 검증 후, 독립된 FastAPI 파서 엔진으로 파일을 전달하고 처리 결과를 반환받아 DB에 저장한다.

- SRS 문서: 6.3 Detailed Interaction Models (문서 업로드 전체 흐름)

## 📋 Task Breakdown
- [ ] Next.js 내 파일 업로드 처리를 위한 Server Action (또는 API Route) 생성
- [ ] 업로드된 파일 형식(PDF, DOCX 등) 및 크기 1차 검증 로직 구현
- [ ] 검증된 바이너리 파일을 FastAPI 엔드포인트(내부 네트워크)로 HTTP POST 멀티파트 전송하는 HTTP Client 로직 작성
- [ ] FastAPI 처리 시작 시 DB의 `PARSED_DOCUMENT` 테이블에 레코드 생성 (상태: `processing`)
- [ ] FastAPI로부터 수신된 결과(정형 JSON, confidence_metrics, highlight_ranges 등) 파싱
- [ ] 결과를 DB의 `DATA_NODE` 테이블에 일괄(Bulk) 저장하고 문서 상태를 `completed`로 업데이트
- [ ] 프론트엔드로 구조화된 데이터 및 하이라이트 메타정보 반환

## ✅ Acceptance Criteria
- **Scenario 1: 정상 파일 업로드 및 FastAPI 연동 성공**
  - **Given** 사용자가 정상적인 법률 서식 파일을 업로드하면
  - **When** Next.js 백엔드가 이를 수신하여 FastAPI로 전달하고 결과를 받으면
  - **Then** DB에 `PARSED_DOCUMENT` 및 연관된 `DATA_NODE` 레코드들이 정상 저장되고, 클라이언트에게 상태 200과 구조화된 JSON 데이터가 반환되어야 한다.
- **Scenario 2: 손상/난독 파일에 대한 에러 처리**
  - **Given** 손상되었거나 파싱이 불가능한 파일이 업로드되었을 때
  - **When** FastAPI에서 에러 응답을 반환하면
  - **Then** DB의 문서 상태가 `failed`로 기록되고 오류 코드 정보가 클라이언트에게 전달되어야 한다.

## 🚧 Constraints
- 통신 규약: 서버 간(Next.js <-> FastAPI) 통신은 외부망을 거치지 않는 Docker 내부 네트워크 기반 통신이어야 한다.
- 메모리 및 타임아웃 제한: 대용량 파일 전송 시 메모리 릭이 발생하지 않도록 스트리밍 방식으로 처리하거나 적절한 버퍼 제한을 설정해야 한다.
