## 🎯 Summary
- 기능명: [BE-02] 하이라이트 셀 덮어쓰기 API (`PATCH /api/v1/nodes/{node_id}`) 및 이벤트 로그 기록
- 목적: AI의 파싱 결과 중 신뢰도가 낮아 하이라이트 된 구간(또는 오류가 있는 구간)의 데이터를 사용자가 수동으로 수정(Override)할 수 있도록 DB 업데이트 엔드포인트를 제공한다.

## 🔗 References (Spec & Context)
- SRS 문서: 3.3 API Overview (Node Override API)
- SRS 문서: 4.1 F2 (REQ-FUNC-008: 하이라이트 셀 수동 덮어쓰기 기능)
- SRS 문서: 6.2 Entity & Data Model (`DATA_NODE` 테이블 구조)

## 📋 Task Breakdown
- [ ] Next.js에 셀 단위 데이터를 수정할 수 있는 `PATCH` 엔드포인트 또는 Server Action 생성
- [ ] 입력받은 `node_id`의 유효성 및 수정할 데이터(`semantic_content`) 검증 페이로드 파서 (Zod 등 활용)
- [ ] Prisma를 사용하여 `DATA_NODE` 테이블의 해당 레코드 식별 및 업데이트 수행
- [ ] 업데이트 시 `overridden_by` (수행자 정보) 및 `overridden_at` (타임스탬프) 필드 동시 갱신
- [ ] 시스템 로그 (로컬 파일 또는 DB 감사 로그)에 덮어쓰기 이벤트 내역 기록 처리 로직

## ✅ Acceptance Criteria
- **Scenario 1: 정상적인 덮어쓰기 완료 및 이력 갱신**
  - **Given** 사용자가 프론트엔드 UI를 통해 특정 셀의 데이터를 수정하여 저장 요청을 보내면
  - **When** 해당 엔드포인트(`PATCH /api/v1/nodes/{node_id}`)가 정상 처리되면
  - **Then** DB의 `semantic_content`가 갱신되고 `overridden_at` 필드에 현재 시각이 기록되며 성공(200) 상태가 반환되어야 한다.
- **Scenario 2: 존재하지 않는 Node ID에 대한 예외 처리**
  - **Given** 잘못된 `node_id`를 포함한 수정 요청이 발생하면
  - **When** 엔드포인트에서 유효성 검증을 수행할 때
  - **Then** HTTP 404 또는 적절한 오류 상태가 반환되고, 잘못된 DB 접근이 차단되어야 한다.

## 🚧 Constraints
- 데이터 무결성: 파싱 문서 전체 상태가 이미 `validated` (검수 승인 완료)인 경우, 노드 덮어쓰기 요청이 거절되도록 상태 검증 로직이 포함되어야 한다. (향후 확장을 고려한 방어 로직)
- 성능: UI 단에서 즉각적인 반영을 위해 응답 지연 시간(Latency)을 최소화해야 한다.
