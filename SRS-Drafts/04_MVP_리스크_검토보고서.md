## 1. 7대 실패 위험 기능 리스트 대비 SRS v0.4 진단

현재 작성된 CorpBrain의 요구사항 명세(SRS v0.4)를 '초보자의 MVP 실행력' 관점에서 냉정하게 평가한 결과입니다.

| 위험 기능 카테고리 | CorpBrain SRS v0.4 내 포함 여부 및 진단 | 위험도 |
| :--- | :--- | :--- |
| **1. 다중 시스템 간 실시간 동기화 필수 구조** | **[포함-주의]** `F4: 단일 클릭 백그라운드 커넥터` (Slack/Github/Notion 증분 동기화, Webhook+Polling 5분 주기). <br>👉 MVP 단계에서 3개 외부 플랫폼의 API Rate Limit과 토큰 만료 예외 처리(401 대응 등)를 모두 구현하는 것은 개발 리소스를 크게 소진시킵니다. | 🟡 높음 |
| **2. 계속 엄밀한 최신 상태를 맞춰야 하는 파이프라인** | **[부분 포함]** `F3: Semantic Dedup` (구버전/중복 문건 자동 배제). <br>👉 백그라운드에서 실시간으로 로컬 임베딩을 돌려 Cosine Similarity를 0.85 기준으로 필터링하는 로직은 파이프라인의 복잡도를 높입니다. | 🟡 높음 |
| **3. 양면시장/복잡한 다중 사용자 운영 플로우** | **[미포함-안전]** 본 시스템은 변호사(검수자)와 CTO(관리자) 중심의 단방향 B2B/B2B2C SaaS(또는 온프레미스) 툴입니다. 양면 시장의 콜드스타트 문제는 없습니다. | 🟢 낮음 |
| **4. 프론트/백/인프라를 과하게 분리한 구조** | **[포함-위험]** `REQ-NF-023: 완전 컨테이너(Docker/K8s) 기반 배포`. <br>👉 초보자가 첫 PoC(A 법무법인 타겟)부터 Kubernetes(K8s) 수준의 오케스트레이션을 로컬 폐쇄망에 플러그앤플레이로 세팅하려는 것은 명백한 오버엔지니어링입니다. | 🔴 매우 높음 |
| **5. 복합 데이터 모델 (Vector DB, Ontology 등)** | **[부분 포함]** RAG 봇 자체는 스코프 아웃되었으나, Dedup을 위해 내부적으로 벡터 임베딩과 JSON, CSV 구조화 파싱이 혼재되어 있습니다. 데이터 정합성 관리가 까다로울 수 있습니다. | 🟡 높음 |
| **6. 다중 에이전트 자동화에 완전 의존하는 설계** | **[미포함-안전]** "비정형 문서 자동 기안/생성 에이전트"를 명시적으로 Out-of-Scope로 배제했으며, 파싱 엔진도 "Rule-based + sLLM" 앙상블로 설계하여 통제력을 갖췄습니다. | 🟢 낮음 |
| **7. 배포 후 운영 자동화 풀패키지 (CI/CD, DevOps)** | **[포함-위험]** `REQ-NF-014: PagerDuty/Slack P1 연동 1분 내 경보`, `REQ-NF-007: 메모리 반환율 100% Graceful Shutdown`. <br>👉 MVP 단계에서 프로덕션 레벨의 모니터링 경보 시스템과 완벽한 메모리 누수 방지까지 요구하는 것은 출시를 영원히 지연시킬 수 있습니다. | 🔴 매우 높음 |

## 2. 진단 요약 및 다온의 현실적 피드백 (Actionable Advice)

회비서님, 현재의 `SRS v0.4`는 **"완성된 상용 엔터프라이즈 제품"**으로서는 훌륭하지만, **"초보자가 1~2주 내에 검증해야 할 MVP(PoC)"** 기준으로는 실패 확률을 높이는 함정(오버엔지니어링)이 곳곳에 숨어 있습니다.

초기 창업자/개발자가 가장 경계해야 할 것은 '기술적 완벽주의로 인한 타임투마켓(Time-to-Market) 지연'입니다. 다음 세 가지를 MVP 스펙에서 과감히 덜어내실 것을 제안합니다.

### 💡 피드백 1: K8s 인프라 및 자동화 모니터링 제거 (가장 시급)
첫 PoC 대상인 'A 법무법인' 하나를 검증하는데 Kubernetes(K8s) 배포 패키징과 PagerDuty 1분 경보 자동화는 필요 없습니다. 
- **수정 제안**: Docker Compose 수준의 단일 노드 배포로 인프라 요구사항을 대폭 하향 조정하고, 알럿 연동(REQ-NF-014)은 스코프에서 제외하십시오.

### 💡 피드백 2: 커넥터 연동 스코프 대폭 축소
Slack, Github, Notion 3사 연동과 실시간 증분 동기화(5분 주기), 토큰 만료 자동 이메일 발송 등은 기능 1개당 1주 이상의 개발이 소요됩니다.
- **수정 제안**: 첫 V0.2 PoC에서는 외부 연동(F4)을 **전면 보류(Out-of-Scope)** 하거나, 철저하게 **'수동 파일 업로드'**만 지원하여 코어 파싱 엔진(F1, F2)의 성능부터 검증해야 합니다.

### 💡 피드백 3: Semantic Dedup의 우선순위 하향
로컬 임베딩을 돌려 Cosine Similarity를 계산하는 것은 벡터 DB 지식이 필요한 무거운 작업입니다.
- **수정 제안**: MVP에서는 이 기능(F3)을 **Should Have에서 Could Have(또는 배제)**로 내리고, 파서(Parser)가 표를 깨지지 않게 추출하는 본질적 가치(북극성 KPI) 증명에 모든 시간을 쏟아야 합니다.

> **Next Step**: 위 피드백을 수용하여 MVP 스펙을 극단적으로 다이어트하시겠습니까? 동의하신다면, SRS v0.4를 수정하여 **[최소 기능 검증용 v0.5]**로 경량화하는 작업을 돕겠습니다.
