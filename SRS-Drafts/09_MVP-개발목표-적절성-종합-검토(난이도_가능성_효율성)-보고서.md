# CorpBrain Lean MVP 종합 검토 보고서 (난이도·가능성·효율성)

## 1. 개요
본 보고서는 5년 차 AI/백엔드 기획자인 회비서님의 배경지식(FastAPI, AWS, ML 튜토리얼 경험)과 '바이브코딩(AI 어시스턴트 주도 개발)' 방식을 전제로, 현재 확정된 Lean MVP 스펙(Next.js 풀스택, SQLite, 100% 로컬망)이 개발 난이도, 구현 가능성, 그리고 시스템/비용 효율성 측면에서 얼마나 적절한지 종합 검토한 결과입니다.

---

## 2. 개발 난이도 및 구현 가능성 검토

### 2.1. 프론트엔드 및 인프라 (난이도: 최하 / 가능성: 최상)
* **스택:** Next.js, Tailwind CSS, shadcn/ui, SQLite, 단일 Docker Compose
* **검토:** AI 코딩 어시스턴트(Cursor 등)가 가장 완벽하게 학습한 글로벌 표준 스택입니다. 회비서님의 프론트엔드 실무 경험 부족을 AI가 100% 상쇄하며, 백엔드 기획 역량을 바탕으로 AI가 생성한 DB 스키마와 API 구조를 완벽히 통제할 수 있습니다. 며칠 내로 프로토타입 UI 구현이 가능합니다.

### 2.2. 백엔드 및 아키텍처 조율 (난이도: 하 / 가능성: 상)
* **검토:** Next.js(Node.js)만으로 모든 것을 처리하기보다, 회비서님에게 가장 익숙한 **FastAPI(Python)**를 파싱 엔진 전용 로컬 마이크로서비스 컨테이너로 분리하여 Docker Compose에 추가하는 것을 강력히 권장합니다. 이는 개발자의 강점(Python)을 극대화하면서 바이브코딩의 장점(Next.js UI)도 살리는 최적의 타협점입니다.

### 2.3. 핵심 파싱 엔진 구현 (난이도: 최상 / 가능성: 중~상)
* **검토:** 로컬 폐쇄망 환경에서 sLLM과 오픈소스 비전 파서(Docling, Surya 등)를 결합해 'TEDS-Struct 99.9%'를 달성하는 것은 AI가 알아서 코딩해줄 수 없는 고도의 엔지니어링 영역입니다. 회비서님의 ML 경험을 이곳에 모두 쏟아부어 Rule-based 로직과 AI 모델을 튜닝하는 데 전체 개발 기간의 80% 이상을 할애해야 합니다.

---

## 3. 시스템 및 비용 효율성 검토

### 3.1. 개발 속도 (MVP 단계의 과도한 시간 소요 회피)
* **평가:** **초고효율 (Extreme Time-to-Market)**
* **검토:** 외부 커넥터(Slack, Notion), 클라우드 DB 연동, K8s 클러스터링 등 초기 MVP에서 시간을 잡아먹는 오버엔지니어링 요소가 PRD v0.6에서 완벽히 제거되었습니다. 바이브코딩 스택과 결합되어 '1~2주 내 PoC 릴리즈'라는 극한의 속도전을 현실적으로 달성할 수 있는 가장 가벼운 구조입니다.

### 3.2. 외부 연동 목표 및 기술 스택 (복잡도/리스크 관리)
* **평가:** **Low Risk, High Focus**
* **검토:** 
  * **외부 연동 제로(0 Byte 아웃바운드):** API 연동 시 발생하는 인증(OAuth), 토큰 만료, 타사 서버 장애, 속도 지연 등의 복잡성을 원천 차단했습니다. 시스템 복잡도가 극도로 낮아집니다.
  * **오픈소스 종속 리스크:** 모든 것을 오픈소스 생태계(로컬 sLLM, 로컬 파서)에 의존해야 하므로, 상용 API(GPT-4o) 대비 초기 파싱 성능이 떨어질 리스크가 존재합니다. 그러나 이를 기획적으로 **'0.5초 하이라이트 UI(인간 검수)'**라는 영리한 UX 장치를 두어 성능 리스크를 시스템적으로 완벽히 상쇄했습니다.

### 3.3. 운영 소요 비용 (클라우드 컴퓨팅 비용 산정)
* **평가:** **클라우드 운영 비용 월 $0 (Zero OPEX)**
* **검토:** 
  * 100% 온프레미스/망분리 제약으로 인해 AWS EC2, RDS, API Gateway 등 **클라우드 컴퓨팅 서비스 비용이 매월 '0원'**입니다. MVP 운영 및 유지보수 측면에서 압도적인 재무적 안정성을 확보합니다.
  * **숨은 허들 (고객사 하드웨어 CAPEX):** 클라우드 유지 비용이 없는 대신, 고객사(법률사무소) 내부에 sLLM을 로컬로 구동할 수 있는 수준의 'GPU 탑재 서버 또는 고성능 워크스테이션'이 반드시 필요합니다. 이는 고객사가 초기 도입 비용(CAPEX)으로 지불하거나, 랩탑/서버 어플라이언스 형태로 묶어 판매하는 하드웨어 비즈니스 모델적 접근이 필요합니다.

---

## 4. 최종 결론
현재의 MVP 스펙과 스택 조합은 **회비서님의 역량(기획+ML+Python)과 한계(프론트엔드), 그리고 바이브코딩의 특성을 완벽하게 조화시킨 최상의 마스터플랜**입니다. 

'클라우드 운영 비용 $0'의 절대적 장점을 무기로 삼되, '고객사의 로컬 GPU 하드웨어 확보'라는 유일한 인프라 허들만 초기 세일즈 과정에서 잘 풀어낸다면 개발과 시장 진입 모두 압도적인 효율과 성공을 거둘 것입니다.
