# 기술 스택 도입 검토 의견 (C-TEC-001 ~ 004)

## 1. 종합 의견
제안해주신 4가지 기술 스택 제약사항(C-TEC) 중 **3개는 현재의 Lean MVP 방향성과 완벽히 일치**하여 적극 도입을 권장합니다. 하지만 **C-TEC-003(Supabase 배포)은 PRD의 핵심 보안 원칙(망분리)을 정면으로 위반할 치명적인 리스크**가 있어 수정이 필요합니다.

## 2. 항목별 상세 검토

### 🟢 적극 도입 권장 (C-TEC-001, 002, 004)
* **C-TEC-001 & 002 (Next.js 단일 풀스택 & Server Actions):**
  * **PRD 부합도:** 100% 일치
  * **사유:** PRD의 핵심 전략 중 하나가 '인프라 복잡도 최소화(단일 Docker Compose 배포)'입니다. 프론트엔드와 백엔드를 Next.js 모노리스로 통합하면 컨테이너 구성이 단순해져 설치형(On-premise) 배포 난이도가 크게 낮아집니다. 또한, 소규모 팀의 MVP 개발 시 컨텍스트 스위칭을 줄여 속도를 극대화할 수 있습니다.
* **C-TEC-004 (Tailwind CSS + shadcn/ui):**
  * **PRD 부합도:** 100% 일치
  * **사유:** MVP의 가장 중요한 덕목인 '개발 속도'를 보장합니다. 특히 AI 코딩 어시스턴트가 가장 잘 훈련되어 있는 스택이므로, 오류나 환각 없이 UI 컴포넌트를 빠르게 찍어낼 수 있는 최적의 선택입니다.

### 🔴 치명적 충돌 및 수정 필요 (C-TEC-003)
* **C-TEC-003 (배포 시 Supabase 사용):**
  * **PRD 부합도:** **정면 위반 (클라우드 의존성 발생)**
  * **충돌 사유:** PRD v0.6의 가장 강력한 제약사항은 **"100% 로컬 망분리 / 아웃바운드 트래픽 0 Byte"**입니다. 배포 환경에서 클라우드 서비스인 Supabase(SaaS)를 연결할 경우, 로펌의 보안 규정(완전 폐쇄망)을 즉각 위반하게 되어 제품 도입 자체가 무산됩니다. (※ Supabase를 로컬 Docker로 Self-hosting 하더라도 MVP 수준에서는 불필요한 인프라 오버엔지니어링입니다.)
  * **✨ 수정 제안 (택 1):**
    1. **극단적 Lean (추천):** `개발 및 배포 모두 로컬 SQLite 사용` (로펌 내부 파싱 작업은 대규모 동시 접속이 없으므로 SQLite 파일 DB로도 충분히 상용 MVP 커버가 가능하며, 백업/이전이 압도적으로 쉽습니다.)
    2. **표준 배포:** `개발은 SQLite, 배포 시에는 Docker Compose 내부에 격리된 단일 PostgreSQL 컨테이너 구성` (클라우드 연결 절대 금지)

## 3. 결론 (SRS 추가 방안)
제안하신 4가지 내용은 기술 스택을 명확히 고정하여 AI 개발의 일관성을 높이는 훌륭한 제약(Constraints)입니다. 

해당 내용들을 **C-TEC-003의 클라우드 의존성을 제거하는 방향으로 수정한 뒤**, SRS의 `1.5 Constraints and Assumptions` 항목 또는 `4.2 Non-Functional Requirements`의 `Deployment` 섹션에 반영하는 것이 구조상 매우 적절합니다.
