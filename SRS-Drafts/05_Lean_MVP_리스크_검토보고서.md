# CorpBrain Lean MVP (SRS v0.5) 리스크 검토 보고서

## 1. 개요
본 보고서는 '초보자로서 실패할 가능성이 높은 MVP 기능 리스트' 7가지를 기준으로, 현재 작성된 **CorpBrain Lean MVP (SRS v0.5)** 사양이 해당 리스크들을 적절히 회피하고 있는지 검토한 결과입니다. 

## 2. 리스크 항목별 사양 검토 (SRS v0.5 기준)

| 실패 가능성이 높은 MVP 기능 (Risk) | CorpBrain SRS v0.5 사양 반영 및 회피 상태 | 검토 결과 |
| :--- | :--- | :--- |
| **1. 다중 시스템 간 실시간 동기화가 필수인 구조** | **Out-of-Scope 처리:** Slack/Github/Notion 등 타사 API 연동 커넥터 기능을 전면 배제 (V0.7 이후로 연기). 수동 파일 업로드 방식으로 시스템 간 동기화 우회 (SRS 1.2, 3.4). | **Pass (안전)** |
| **2. 계속 엄밀한 최신 상태를 맞춰야 하는 외부 데이터 파이프라인** | **Out-of-Scope 처리:** 외부 데이터 수집 파이프라인(Ingestion) 및 실시간 동기화를 삭제. 사용자가 직접 필요 시점에 업로드하는 방식(On-demand)으로 상태 동기화 관리 부담 제거. | **Pass (안전)** |
| **3. 양면시장/복잡한 다중 사용자 시나리오의 운영 및 관리 플로우** | **단일 페르소나 집중:** '스타트업 CTO' 등 다중 사용자 시나리오를 제거하고, 오직 '이지언(부티크 로펌 대표)' 단일 페르소나의 '수작업 검수 시간 단축'이라는 선형적인 단일 플로우에 집중함 (SRS 2. Stakeholders). | **Pass (안전)** |
| **4. 프론트/백/인프라를 과하게 분리한 구조** | **단순화된 배포 구조:** K8s 클러스터링이나 MSA를 배제하고, 단일 서버에서 구동 가능한 단일 Docker Compose 기반 배포로 인프라 복잡도를 극소화함 (SRS 4.2 REQ-NF-009). | **Pass (안전)** |
| **5. 온톨로지나 Vector DB 기반의 RAG, 기타 SQL + NoSQL 이 복합 사용되는 데이터 모델** | **오버엔지니어링 배제:** 로컬 임베딩 기반 Semantic Dedup 및 Vector DB 연동을 초기 스펙에서 완전히 덜어냄. 관계형 데이터(RDB) 수준의 단순 ERD만 유지함 (SRS 1.2, 6.2). | **Pass (안전)** |
| **6. 특정 기능이 다중 에이전트 자동화에 완전히 의존하여 제공되는 AI 중심 설계** | **인간 제어 중심(Human-in-the-loop):** 비정형 문서 자동 기안 에이전트를 배제함. AI는 오직 파싱과 'Confidence 하이라이트(시각화)'만 보조하고, 최종 승인 및 수정(Override)은 인간 검수자(Validator)가 수행하도록 통제력을 유지함 (SRS 4.1 REQ-FUNC-008). | **Pass (안전)** |
| **7. 배포 후 운영 자동화까지 한 번에 하려는 구조 (DevOps 풀패키지)** | **모니터링 최적화:** PagerDuty 연동 및 실시간 Alert 발송 등 복잡한 운영 자동화 삭제. 엔진 크래시 시 로컬 로그 기록과 Docker 자체의 `restart_policy`에만 의존하여 복구 복잡도를 대폭 낮춤 (SRS 4.2 REQ-NF-011, 012). | **Pass (안전)** |

## 3. 종합 평가 및 제언

현재 작성된 **CorpBrain SRS v0.5**는 초보 창업자 및 초기 개발 팀이 겪기 쉬운 **'오버엔지니어링' 및 '기능 과적(Scope Creep)'의 함정을 완벽하게 회피**하고 있습니다. 
가장 리스크가 큰 외부 연동, 자동화, 분산 시스템, 다중 에이전트 요소를 모두 과감히 잘라내고, **"수동 업로드 -> 로컬 sLLM 기반 파싱 -> 하이라이터 UI를 통한 인간 검수"** 라는 매우 직관적이고 통제 가능한 1차원적 선형 플로우만 남겼습니다.

**💡 향후 개발을 위한 추가 제언:**
1. **MVP 릴리즈 주기 최단축:** 현재 스펙은 아키텍처와 기술적 복잡도가 매우 낮아졌으므로, V0.2 PoC(A법무법인 계약서 1종 파싱) 릴리즈를 1~2주 내에 끝내는 것을 최우선 목표로 삼아 실행력을 극대화해야 합니다.
2. **AI 파싱 엔진의 코어 퀄리티 집중:** 시스템 복잡도가 대폭 줄어든 대신, 사용자가 체감하는 유일한 핵심 가치는 '파싱 정확도(TEDS-Struct 99.9점 유지)'가 되었습니다. 따라서 백엔드/인프라 개발에 쓸 리소스를 'Dirty Form Dataset'(손상되거나 복잡한 표) 확보와 프롬프트/파싱 룰 튜닝에 전량 투입하는 전략이 필요합니다.
