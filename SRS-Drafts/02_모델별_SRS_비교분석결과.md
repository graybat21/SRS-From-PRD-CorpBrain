# Gemini vs Opus 기반 SRS 생성 결과 비교 분석

## 1. 개요
동일한 PRD와 프롬프트를 기반으로 작성된 두 SRS 문서(`SRS_v0.1_gemini.md`와 `SRS_v0.1_opus.md`)의 차이를 분석한 결과, **요구사항의 분해 해상도(Granularity)**와 **문서의 구조화(Table 활용)** 측면에서 극적인 차이가 확인되었습니다. 

## 2. 정량적 지표 비교

| 비교 항목 | SRS_v0.1_gemini.md (Gemini) | SRS_v0.1_opus.md (Opus) | 차이점 요약 |
| :--- | :--- | :--- | :--- |
| **파일 크기** | 약 18 KB (211 라인) | 약 33.5 KB (538 라인) | Opus가 약 2배 이상의 분량 생성 |
| **기능 요구사항(FUNC)** | 8개 | **20개** | Opus가 Feature별로 더 잘게 쪼개어(Atomic) 분석함 |
| **비기능 요구사항(NFR)** | 12개 | **21개** | Opus가 Category(보안, 성능, 모니터링 등)를 세분화함 |
| **API Endpoint 정의** | 4개 | **11개** | Opus가 예외 처리 및 검수 로직용 API까지 구체화함 |
| **Mermaid 다이어그램** | 2개 | 4개 | Opus가 프로세스를 2개의 흐름(검수/백그라운드 연동)으로 분리 |

## 3. 항목별 심층 비교 분석

### 3.1 작성 스타일 및 표(Table) 활용도
- **Gemini**: 대부분의 항목(이해관계자, 제약사항, 외부 시스템 등)을 **글머리 기호(Bullet points)** 형태로 서술하여 가독성은 좋으나 상세한 메타데이터가 부족했습니다.
- **Opus**: 프롬프트의 "표 사용 적극 권장" 지시를 극단적으로 수용했습니다. 용어 정의, 제약사항, 이해관계자, 시스템 인터페이스, API, 데이터 모델 등 **거의 모든 섹션을 Markdown 표로 구조화**하여 문서의 정보 밀도와 규격화 수준을 높였습니다.

### 3.2 요구사항 분해의 해상도 (Atomic Requirement)
- **Gemini**: 1개의 기능(Feature)을 1~2개의 요구사항으로 크게 묶어서 처리했습니다. (예: F1 기능을 2개 요구사항으로 정리)
- **Opus**: 1개의 기능을 예외 케이스(File Size 제한, Timeout 등)와 관리자/사용자 액션(복원, 덮어쓰기 등)까지 세분화하여 하나의 기능당 3~4개의 요구사항으로 **원자적(Atomic) 분해**를 수행했습니다. 이는 실제 개발 및 QA(Test Case 작성)에 훨씬 유리한 구조입니다.

### 3.3 기술적 구체성 및 시스템 설계 역량
- **Gemini**: PRD에 명시된 내용(예: POST /clean 등)을 그대로 옮겨 적는 수준의 충실한 요약자 역할을 수행했습니다. 전체 엔터티를 하나의 표에 묶어 표현했습니다.
- **Opus**: Senior Requirements Engineer라는 페르소나에 깊이 몰입하여, PRD에 없는 기술적 디테일(예: `PATCH /api/v1/nodes/{id}` 수동 덮어쓰기, `GET /api/v1/dedup/excluded` 배제 문서 조회 등)을 **스스로 추론하여 아키텍처 빈틈을 메웠습니다.** 엔터티(DB 테이블) 모델링도 테이블별로 PK/FK, Enum 제약 조건까지 명시하여 스키마 설계에 근접한 결과를 냈습니다.

### 3.4 추적성(Traceability) 구현
- **Gemini**: Story 단위로 뭉뚱그려 추적 매트릭스를 작성했습니다.
- **Opus**: 개별 Acceptance Criteria(AC) 단위까지 쪼개어 해당 AC가 어떤 FUNC ID와 NFR ID로 연결되고, 어떤 Test Case 시나리오로 이어지는지 1:1 매핑을 명확히 수행했습니다.

## 4. 결론 (회비서 관점의 시사점)

1. **Gemini 모델**은 **비즈니스 기획자(PM/PO)가 한눈에 시스템의 개요와 큰 그림을 파악**하고 빠르게 리뷰하기에 적합한 "요약 중심의 명세서"를 작성했습니다.
2. **Opus 모델**은 실제 **개발자 및 QA 엔지니어에게 바로 전달해도 무방할 수준의 "구현 중심의 명세서(Detailed Design에 근접)"**를 작성했습니다.
3. 향후 사업기획 및 제품 설계 시, **초기 아이데이션이나 경영진 보고용으로는 Gemini의 포맷**이, **실제 스프린트 백로그 구성 및 외주/개발팀 이관용으로는 Opus의 포맷**이 압도적으로 유리합니다.
